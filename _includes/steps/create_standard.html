Ideate a new standard

First of all, you may want to explore what standards exist at the moment before going ahead and create one. Just in case there is one already for the topic you want to propose which could get yours not approved to be worked on.

Other than that, feel free to think of any kind of standard that would make your life and others' around you easier. A standard is going to be from a basic to advanced set of rules, some optional may exist, that can be easily verifiable so that when others apply it on their project and ask for compliance validation, maintainers of the standard can verify the project actually follows the standard.

Submit it to the meta standard

Once you have a solid idea for a standard that does not exist yet, you can go ahead and submit it to the meta standard as a Github Issue. Here is a shortcut link to do that: Submit Issue to meta standard.

Note that you will be asked for the following information:

  1. What is the purpose of this Issue?: In this case you are proposing a new standard so just delete the other options.
  2. Describe the purpose of this issue a bit further: You may now describe the standard you are proposing to create. You can also include links with information about other standards out there that are serving as inspiration. Anything related to why the standard should exist so that others can engage on discussing it till it reaches a good state where the Standards maintainers, which are the meta standard maintainers, can give you the green light for creating such a standard.

Invite people into the discussion

It's really important that the community takes a huge part of creating new standards as it's the community that is going to make them their own, evolve them and work with them. There is no use of a standard proposed, created and used by a single person. We need various points of views, different knowledge put together to construct the way we all want to work with each other.

For that reason, is vital that you invite more people into the submitted standard. Just mention them on a comment typing "@" for which their name will be suggested if they have a GitHub user, asking for their opinion. Ask for resources of existent external standards to base on in order to pick the best from them to create the best standard possible.

Get your new standard approved by Standards maintainers

Once you have a very fruitful standard discussion where resources where shared and a clear roadmap of how a standard should be written so it benefits everyone, you can mention @Standards/maintainers to bring the maintainers attention and to ask for approval.

This approval request is just to get feedback about what was described in the last 3 steps. If you have an unique standard proposal with a healthy discussion and people involved, you will get approval. This step is important to avoid creating a bunch of similar standards or standards without community involvement.

Start working on the standard definition

You are all set to work on the definition of the standard. You will need to follow the meta standard which is the standard for creating standards.

You can work on the new standard within your own profile creating a repository or even within an organization you manage. The important thing is that you create a repository and follow the meta standard.

Get your standard validated for its publication

The final step comes when you worked on the standard following the meta standard and you are satisfied with the results. In this step, you will submit a new issue in the meta standard but this time requesting validation.

In the description you will need to include a link to the repository that constitutes the new standard you worked on so that Standard maintainers can check that it follows the meta standard.

If everything needs to be modified/added, maintainers will give the appropriate feedback through that submitted issue. If everything looks good, you will be asked to:

  1. create a new release for your standard to start versioning its changes;
  2. include a badge in the README file, like the one the meta standard has pointing to the correct version that was followed;
  3. try to set the correct language of the repository to be able to categorize the standard;
  4. and to give permission to one of the maintainers to manage your repository so it can be moved to the Standards organization.

Once that's done, it will start showing up on the explore section of the homepage, searchable and ready to be applied by others. Don't forget to ask for compliance for your projects to show they follow this new standard.
